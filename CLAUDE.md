# CLAUDE.md — BookChars

Поведенческие правила для работы с кодовой базой BookChars.
Основано на принципах Андрея Карпатого (forrestchang/andrej-karpathy-skills), адаптировано под проект.

**Стек:** Next.js 14, Claude API (Anthropic), Pollinations.ai, Supabase, Vercel

---

## 1. Думай до кода, не после

**Не угадывай. Не молчи о неопределённости. Называй предположения вслух.**

Перед реализацией:
- Явно назови все предположения. Если не уверен — спроси.
- Если задача допускает несколько интерпретаций — перечисли их, не выбирай молча.
- Если есть более простой способ — скажи об этом.
- Если что-то непонятно — остановись и спроси, не продолжай наугад.

---

## 2. Доводи до конца

**Задача считается выполненной только когда она полностью работает.**

- Не останавливайся на полпути — если начал функцию, доведи её до рабочего состояния.
- Не оставляй `// TODO`, `// заглушка`, `// реализовать позже` без явного согласования.
- Если задача большая — раздели на шаги и сообщи план ДО начала реализации.
- После завершения — проверь что всё связано: импорты, экспорты, вызовы функций.

План для многошаговых задач:
```
1. [Шаг] → проверка: [что должно работать]
2. [Шаг] → проверка: [что должно работать]
3. [Шаг] → проверка: [что должно работать]
```

---

## 3. Не трогай то, что работает

**Изменяй только то, о чём тебя попросили. Чужой рабочий код — не трогать.**

При редактировании существующего кода:
- Не «улучшай» соседние функции, компоненты, стили — если их не просили.
- Не рефакторь то, что не сломано.
- Не меняй форматирование и комментарии в чужих блоках.
- Если заметил проблему рядом — упомяни словами, но не правь сам.

Тест: каждая изменённая строка должна напрямую относиться к запросу.

---

## 4. Только реальное API — никаких выдумок

**Используй только методы и параметры, которые реально существуют.**

### Supabase
- Проверяй актуальную документацию перед использованием методов.
- Не выдумывай методы клиента — используй только то, что есть в `@supabase/supabase-js`.
- RLS политики — не предполагай их наличие/отсутствие, уточняй.
- Типы генерируй из реальной схемы БД, не придумывай структуру таблиц.

### Anthropic Claude API
- Используй только актуальные параметры из официальной документации.
- Модели: `claude-sonnet-4-6`, `claude-haiku-4-5-20251001` — не придумывай названия моделей.
- Не выдумывай параметры запроса — проверяй схему `messages`, `system`, `max_tokens`.

### Pollinations.ai
- Используй только задокументированные эндпоинты и параметры генерации.
- Не предполагай наличие методов — проверяй актуальный API.

### Next.js 14
- App Router — не смешивай с Pages Router.
- Server Components vs Client Components — явно разграничивай, не угадывай.
- Не используй устаревшие паттерны (`getServerSideProps`, `getStaticProps` в App Router).

---

## 5. Минимализм

**Минимум кода, который решает задачу. Ничего лишнего.**

- Никаких фич сверх запрошенного.
- Никаких абстракций «на будущее» без запроса.
- Никакой «гибкости» и «расширяемости» без явной необходимости.
- Если написал 150 строк, а можно 40 — перепиши.

---

## Признаки что правила работают

- В диффе нет изменений, которые не относятся к задаче
- Нет заглушек и незаконченных функций
- Нет вызовов несуществующих методов
- Уточняющие вопросы задаются ДО реализации, а не после ошибок
